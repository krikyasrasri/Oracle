LCS Database Support

 

LCS means Life Cycle Server.

 

The LCS is the reporting system for the RSC. It contains RSC transaction data. The data is retained for longer in the LCS than it is in the RSC. The LCS is used for many reporting functions vital to the APC. It also includes MIGS transactions as these transactions go through the RSC.

 

This web page contains information to assist with DBA support of this environment.

Overview

The LCS production environment comprises two LCS servers. These two servers are linked by replication to the 3 RSC servers. This document applies to the two LCS servers only. For information on the RSC servers refer to the RSC Support Handbook.

The two LCS production servers are:

Database Name

Physical Location

Description

lcs2syd1

plcs1

Sydney North Ryde

Primary LCS server

lcs2ksc0

plcs2k

Kansas City

DR or failover LCS server

Both primary and DR are fully operational standalone databases, identically configured. Both plcs1 and plcs2k should be the same in terms of database size and overall configuration. The same processing is done on each, except for EMS processing which only takes place on plcs1. There is no DR for EMS and EMS is never pointed at plcs2k, although the plcs2k database is fully capable of supporting EMS and is configured to do so.

The LCS is used primarily for reporting. A wide range of essential data extraction and reporting functions are performed by the LCS. See the LCS applications section for information on the various LCS applications.

Criticality

LCS is currently used for essential MIGS reporting only. These MIGS reports are highly critical and time sensitive. The most important MIGS reports are Draft Capture Files. These reports run throughout the day, at specific times according to the requirements of each customer. The reports are sent to MasterCard customers and contain financial data for settlement, and therefore this reporting function is considered highly critical.

Outages of LCS can be taken whenever Draft Capture Files are not being run, and on most days there is a window of at least 2-3 hours in the late morning Sydney time when planned outages can be taken. See Procedures for Taking a Planned Outage.

Master and HOT1

From the application perspective, the two LCS production servers lcs2syd1 and lcs2ksc0 operate in one of two modes, master or hot1. From the DBA perspective there is no difference between these two modes, however the terminology is widely used by application support and operations teams. At all times, one of the servers is designated as Master, and the other is HOT1. Never are both Master or both HOT1 at the same time. Nominally, lcs2syd1/plcs1 should be master most of the time, although lcs2ksc0/plcs2k can also be master for extended periods and not just during contingency exercises.

Master

The Master LCS server is the one from which all reports are sent. Both LCS servers will always generate the same sets of reports independently of each other but only one of these servers will be the one from which the reports are actually used. Reports generated by an LCS are normally transferred to members via a process called GFT. This process involves sending reports from the LCS server to a mainframe. Some reports are manually copied off the LCS server by UNIX Operations. These various report distribution functions are carried out on the Master LCS server only. UNIX Operations know which server is master. From the database perspective, there is no difference, as whether or not an LCS database is master or HOT1 does not impact the database in any way.

HOT1

Whichever of the two LCS servers is not master is called HOT1. The HOT1 LCS server does exactly the same processing as the master LCS server except that no reports are transmitted copied or sent from the HOT1 LCS server. This is done from the master LCS only. From the database perspective, however, the HOT1 LCS is indistinguishable from the Master LCS the exact same processing occurs on the database regardless of which is which.

Failovers

UNIX Ops can failover LCS between master and hot1 periodically as needed. This is a trivial exercise and not important to the DBA. A failover is simply setting a flag to indicate which server reports are transmitted from. An LCS failover is completely transparent to the DBA. Failovers are done by the OCC at the request of the RSC/MIGS Support teams.

Replication

There are two forms of database replication implemented in the LCS production environment:

  Replication of the OTHER_TXN_LOG table one way from each RSC database to each LCS database. This replication is being done by Oracle Streams.

  Replication of the MIGS_LCS_DATA table one way from each MIGS database to each LCS database. This replication is being done by Oracle Streams.

There is no replication of any kind from any LCS database to any other database, not even other LCS databases. All LCS replication is one-way to each LCS database from a non-LCS database.

Space Management

Adding Datafiles to the Database

If adding space to the database you need to take into consideration the fact that the databases are on raw slices. Here is the procedure for adding a datafile to a tablespace on any LCS database

         Run vxprint to identify free disk slices that can be used for the new datafile. You can also check the disk configuration spreadsheets for lcs2syd1 and lcs2ksc0. Free slices can be identified by their name which will all begin with free_.

Free slices that are reserved for additional redo logs will be named free_redo_<thread>_<group><member>. eg: free_redo_01_07a.

Free slices that are reserved for specific tablespaces will be named free_<tablespace name>_nn, eg: free_lcs_idx_15.

         Use sesudo vxedit-rename to rename a free disk slice. For example:

sesudo vxedit-rename d plcs1 free_lcs_idx_15 lcs_idx_15

         Add a datafile to the tablespace using the ALTER TABLESPACE ADD DATAFILE command.

         Make sure you do the same on both LCS databases, not just one.

         Send an email or raise a DA ticket for the primary DBA that describes what slice was renamed and the exact syntax of the ALTER TABLESPACE command that was run.

Adding New Raw Slices

Occasionally (only a few times per year) new free disk slices may need to be added to the plcs1 and plcs2k disk groups. This is done by UTS Sydney (John Sugar). Here is what you need to do to get new raw slices added to the raw disk group.

         Use the command vxprint v|grep free to see what free disk slices are available, and from this output decide whether you really need to create any more disk slices.

         Open up the disk slicing spreadsheets for each LCS production server.

         Decide what sized disk slices need to be added. This will depend on what datafiles need to be added. The size of the disk slice should always be 1Mb higher than the size of the Oracle datafile that will reside on that disk slice. So, for example, if you need to add some disk slices for the LCS_IDX and LCS_TBL tablespaces, you would need to create disk slices that are 20001Mb in size. Below are the most common sized slices that would normally need to be added for the LCS environment:

501M

1001M

2001M

5001M

20001M

 

         Using the disk slicing spreadsheets, work out what set of disks the new volumes will be created on. For LCS, the configurations of lcs2syd1 and lcs2ksc0 are different:

lcs2syd1

 

On lcs2syd1 we have 15 x 100Gb LUNs for the database. These are striped into 5 sets of 3 x 100Gb LUNs. You will see from the spreadsheets that each set of 3 x 100Gb LUNs are denoted Stripe A, B, C, D and E. As at May 2010, stripes A, B, C and D are full. So in the case of any new slices, the new volumes will need to be created on stripe E.

lcs2ksc0

 

On lcs2ksc0 we have 12 x 136Gb LUNs for the database. These are striped into 3 sets of 4 x 100Gb LUNs. You will see from the spreadsheets that each set of 4 x 100Gb LUNs are denoted Stripe A, B and C. As at May 2010, stripes A and B are full. So in the case of any new slices, the new volumes will need to be created on stripe C.

 

         Modify the script 01_createvol.sh as needed, depending on the number and names of the new disk slices that are needed, and you need to include the letter denoting the stripe (E for lcs2syd1 and C for lcs2ksc0).

         Place your modified copy of 01_createvol.sh along with 00_functions.sh in a directory on each database server. The 00_functions.sh script must be present in that directory but is never run directly. You should not need to modify 00_functions.sh because it is written to work on every LCS database server that uses raw slices.

         Have UTS cd into the directory that contains these files and then run 01_createvol.sh as root.

Data Retention Policy

In LCS the retention policy for credit transactions is different than for debit transactions. As at March 2009 the retention policy for LCS in production is:

  All transaction data is retained for 60 days.

  After 60 days all 0100 transactions are purged.

  All remaining transactions are purged after 180 days.

  MIGS transactions in the ARISTION_EXTRACT table are purged after 7 days

In other words, the LCS retention policy is 60 days for credit and 180 days for debit.

 

BLOB data, that is, the data in the OTHER_TXN_LOG table is retained for 3-4 days.

 

The retention policy may be different in dev or test systems. You can determine the retention policy for any LCS database as follows:

 

  The number of days retention for the data in the OTHER_TXN_LOG table is given by the following query:

 

SELECT param_value

FROM partition_param

WHERE param_name = LOBMIN_KEEP;

 

  The number of days retention for credits (0100 transactions) is given by the argument to the rsc_purge.purge_credit_older() job:

 

SELECT what

FROM dba_jobs

WHERE schema_user = 'ATS_DB'

AND what LIKE 'rsc_purge.purge_credit_older%';

 

  The number of days retention for all remaining transactions (everything except 0100 transactions) is given by the argument to the rsc_purge.drop_older() job:

 

SELECT what

FROM dba_jobs

WHERE schema_user = 'ATS_DB'

AND what LIKE 'rsc_purge.drop_older%';

  The number of days retention for partitions of the ARISTION_EXTRACT table are given by the argument to the pkgAristion.DropPartitionsOlder() job:

 

SELECT what

FROM dba_jobs

WHERE schema_user = 'ATS_DB'

AND what LIKE ' pkgAristion.DropPartitionsOlder%';

 

The retention policy is enforced by the following stored procedures which are executed daily by DBMS jobs that run just after midnight:

 

rsc_txnlob.drop_old_days drops partitions older than the retention policy from the OTHER_TXN_LOG table.

rsc_purge.purge_credit_older deletes all 0100 transactions older than the retention policy.

rsc_purge.drop_older drops partitions older than the retention policy from the LifeCycle tables.

pkgAristion.DropPartitionsOlder drops partitions older than the retention policy from the ARISTION_EXTRACT table.

 

For details on these jobs, see the Daily Partition Maintenance Jobs and the Other Scheduled Jobs sections.

Changing the Retention Policy

If you need to change the retention policy on any LCS database you can do this as follows:

  To change the retention policy for the OTHER_TXN_LOG table run the following SQL:

UPDATE partition_param

SET param_value = <days>

WHERE param_name = LOBMIN_KEEP;

 

  To change the retention policy for credit transactions, modify the DBMS job in the ATS_DB schema that calls the rsc_purge.purge_credit_older stored procedure.

  To change the retention policy for all other transactions, modify the DBMS job in the ATS_DB schema that calls the rsc_purge.drop_older stored procedure.

  To change the retention policy for the ARISTION_EXTRACT table, modify the DBMS job in the ATS_DB schema that calls the pkgAristion.DropPartitionsOlder stored procedure.

Job run time

What

00:15 daily

<user_dump_dest>/purge.log

Purpose: This job purges credit transactions (0100 transactions) older than the retention policy from the 6 tables whose name begins with LC_TXN%. This job operates on the LifeCycle tables only. It does this by manipulating partitions for optimal performance and space reclamation. Here is the logic that this job uses:

  Insert the primary key of all transactions that need to be kept (any transaction that is not an 0100) into the TXN_KEEP table.

  Exchange these 6 temporary tables with the corresponding partitions of the LifeCycle tables that are being purged.

  Drop the 6 temporary tables.

Optimizer Statistics

The Oracle 10 optimizer requires good statistics. In the RSC and LCS environments the default GATHER_STATS_JOB that is installed as part of the Oracle 10 upgrade has been disabled. This is because it doesnt work in the RSC and LCS environment. We have customized jobs which gather and copy optimizer statistics in a manner appropriate to the needs of the RSC and LCS environment.

This section describes in detail how optimizer statistics are maintained in all LCS databases.

In the LCS environment we gather statistics at the partition level for the LifeCycle tables but not the BLOB tables. We also gather statistics for all other non-partitioned tables plus we gather global stats in addition to the partition level stats on the LifeCycle tables. Thus we may consider there being two categories of statistics gathering on the LCS platform and I will now discuss each of these categories separately.

Referential Integrity Constraints

Here is some information about the referential integrity constraints in the LCS schema:

  For performance reasons there is no referential integrity enforced by the database between the 6 LifeCycle tables, nor from any static table (non-LifeCycle table) to any LifeCycle table.

  The static tables (that is, all the other tables except for the LifeCycle tables and any table replicated by DA replication) have a lot of referential integrity enforced by the database. The number of RI constraints in the database is quite high considering the number of tables.

  All referential integrity constraints in the database should be enabled during normal operation. Unlike the RSC, which has the legacy PARMS application whereby some constraints are always disabled, LCS has no such application and all constraints should be enabled.

Exporting Production Data to a Dev or Test Environment

Occasionally RSC Support may ask for production data to be copied to a development or test environment. It is important to distinguish two types of requests:

  The first and most common type of request is a request to restore historical transaction data (LifeCycle tables only) to the MTF system. This process is fully documented on a separate web page. Please see my document titled Loading Historical Data into LCS. For these types of requests, transaction data that has been archived from the production RSC into a TAR file is restored into the tlcs1a database on rsc1syd2 so that application teams can rerun reports or Draft Capture Files that have been requested by members. Please refer to this separate documentation if you are requested to perform this type of data migration request.

  The second type of request is to restore static data (that is, data from configuration tables not transaction tables) into one of the dev or test systems. Such a request would need to be accompanied by a dev/test CCF that has been approved by Information Security if the data is being moved off the production network. Because the MTF system (rsc1syd2) is on the production network a CCF is not needed if restoring from production to rsc1syd2. But if restoring from production to any development server then a CCF is needed.

In the remainder of this section I discuss the procedure for performing the second type of request to restore static data from production to dev or test. If transaction data is being restored, then please refer to Loading Historical Data into LCS

Note that the request may also take the form of a request to copy static data from one test system to another test system instead of copying from prod to test. Either way it doesnt matter, the procedure is the same.

Ok, heres what to do.

Step 1: Identify and export the static tables to be exported

First you need to generate an export parameter file on the system that you are exporting from. This export parameter file should contain the list of tables that are being exported. For most export requests the list of tables would be the static configuration tables. These are all tables in the ATS_DB schema except for the following:

         LifeCycle tables

         BLOB tables

         Core Reports tables

         Miscellaneous summary tables

         SAM tables

         DCF Scheduler tables

         KitMon tables

 

You can run selectstaticnoncrtablesforexport.sql on the source database to generate an export parameter file. This script will create an export parameter file called staticnoncrtablesLCS.parfile. You can then use this parameter file to run the export. Perform the export as the schema owner, ATS_DB. Youll need to sesudo ats_db to do this.

Not every request to export data will be for the same tables. Sometimes you may be asked to export a different subset of the tables. It will normally be one of the subsets in the above list, such as the Core Reports tables, SAM tables, etc. Customize your export parameter file appropriately.

Step 2: Copy export file to target system

Transfer the export file to the target system.

Step 3: Disable constraints, truncate tables, import, then enable

It is important to carry out each of the steps below exactly as described. Do the following on the target database, that is, the database that you are copying the data into:

  Disable all foreign key constraints in the ATS_DB schema.

  Truncate all the tables that you exported. These are the same tables that are listed in the export parameter file that you generated in Step 1.

  Import the data into the ATS_DB schema. Perform the import as the schema owner, ATS_DB.

  Enable all foreign key constraints in the ATS_DB schema.

This completes the restore request.

Information on some LCS Applications

Here is an overview of some LCS applications to which DBA attention may need to be directed from time to time.

mc_txnblob_read

mc_txnblob_read is the component of the application that reads the BLOB tables and populates the LifeCycle tables with transaction data. It is active on every RSC and LCS server except for the RSC master. Remember that the BLOB tables are the only tables that are replicated from the RSC master to every other RSC and LCS in the environment. The LifeCycle tables themselves are not replicated. So on every RSC and LCS database except for the master, the only way that the LifeCycle tables get populated is by mc_txnblob_read.

What mc_txnblob_read does is look for new rows arriving in the OTHER_TXN_LOG table by replication. If replication has inserted a new row into OTHER_TXN_LOG then mc_txnblob_read will read the BLOB column in that row and use the data stored in the BLOB to insert transactions into the LifeCycle tables. Therefore mc_txnblob_read is closely tied with replication and is an essential piece that is needed in order for the LifeCycle tables to be populated with data on LCS and the non-master RSCs.

For more information see the section on replication customizations in the RSC Support Handbook.

Things to be aware of about mc_txnblob_read are:

  There is a Streams DML handler called INSERT_TRK_DML_HANDLER which causes a row to be inserted into the OTHER_TXN_TRK table whenever a Streams apply process inserts a row into OTHER_TXN_LOG. See the section on replication customizations for details. mc_txnblob_read polls the OTHER_TXN_TRK table. So whenever the DML handler fires and inserts a row into the OTHER_TXN_TRK table, mc_txnblob_read knows that it has to process that row and populate the LifeCycle tables. After successfully processing a row, mc_txnblob_read deletes the row from OTHER_TXN_TRK.

  mc_txnblob_read does not delete rows from the OTHER_TXN_LOG table. This is done by the daily partition maintenance job rsc_txnlob.drop_old_days.

  mc_txnblob_read can be shutdown independently of the rest of the LCS applications. This can be done by UNIX Operations.

  There is a Tivoli monitor for mc_txnblob_read so if it is shutdown manually or stops for any other reason, the Tivoli job will automatically restart it. If mc_txnblob_read needs to be taken down for a specific reason such as a CCF, then UNIX Operations need to disable the Tivoli monitor as well as stopping the process. They can do this using the OPS GUI.

  If there is ever any risk that replication may replicate rows of the OTHER_TXN_LOG table out of sequence then it is essential that mc_txnblob_read is shutdown on all nodes except for the RSC master before this happens. Replication would only ever replicate out of order if rows in the OTHER_TXN_LOG table are inserted out of order, causing the DML handler that populates the OTHER_TXN_TRK table to fire out of order. This might happen if Streams replication is deinstalled and reinstalled. It might also happen if rows of the OTHER_TXN_LOG table are ever manually copied from one database to another, or if rows in the OTHER_TXN_TRK table are manually loaded. This should never happen under normal circumstances, but if it does, it is essential that mc_txnblob_read be shutdown before any such activity occurs!

  Poor performance of a query by mc_txnblob_read was observed on LCS after upgrading to Oracle 10 in production. This was rectified by generating a stored outline for the poor performing query in the ATS_DB schema. Stored outlines must be enabled in both LCS production databases to ensure continued good performance of mc_txnblob_read.

mc_txnblob_read connects to the database as the switch user (RSCPROD, RSCTEST, RSCDEV etc).

mc_txnblob_read is supported by RSC Support.

prod_fix

RETIRED EARLY 2011 NO LONGER USED!!

The name prod_fix refers to a shell script called /apps_01/prod_fix/bin/prod_fix.ksh that runs several times per day on every LCS server. The purpose of this script is to correct data issues with transactions stored in the LifeCycle tables. It selectively updates, inserts and deletes data in the LC_TXN_DELTA, LC_TXN_AMTS and LC_TXN_ITEMS tables to correct issues with the data. This program is supported by RSC Support.

The DBA needs to be aware of this script as it is critical to the successful operation of the LCS environment and must perform adequately.

It is scheduled to run every 2 hours starting at 12:15 and continuing until 00:15. It does not run between 00:15 and 12:15. It processes all transactions since it last successfully ran. So if for some reason prod_fix hasnt run at its scheduled time or if one or more scheduled runs have been missed, then when it does run it will attempt to process everything not yet processed. This can potentially cause rollback issues because the script only has one commit at the very end, when all data has been processed. We have never experienced issues with undo_retention or capacity of the undo tablespace caused by prod_fix but this is something to be aware of. If for example, one of the SQL statements in prod_fix began to perform poorly, it may cause the script to run for longer than it should which in turn may cause an ORA-01555. The problem would be compounded if the script fails, because then next time it runs it would have even more data to process, making matters worse. If this ever happens, then RSC Support should run prod_fix manually, specifying a narrower start and end date range so that the script is run multiple times for a smaller date range rather than trying to process a huge body of data in one hit.

Prod_fix has to run before core reports and other jobs will run. Core reports will not run if prod_fix has not processed the data that it needs. Prod_fix is also a prerequisite for the monthly billing report, and the DCF Scheduler has a configurable wait for prod_fix flag which, if turned on, will cause the DCF Scheduler to wait for prod_fix to be up to date before running a DCF.

Prod_fix writes to a table called PROD_FIX_LOG which contains a row for each time prod_fix runs, and records the date range of transactions over which prod_fix has run. This is how prod_fix and other applications in LCS know where prod_fix is up to.

Prod_fix is supported by RSC Support.

RSC Count Script

All RSC and LCS databases in the replicated environment should contain the same number of transactions. That is, the following query should return the same value on all RSC and LCS databases in the environment:

SELECT COUNT(*)

FROM lc_txn_data

WHERE log_datetime >= TO_DATE(<yesterdays date>,YYYYMMDD)

AND log_datetime < TO_DATE(<yesterdays date>,YYYYMMDD) + 1;

 

In the production environment there is a script that runs once per day to verify that the count of transactions is the same on all databases in the environment. This script runs on lcs2syd1 and lcs2kcs0. The script runs the above query on the local LCS database and on all 3 remote RSC databases, and then checks that the output is the same from each database. So:

 

  The script on lcs2syd1 runs the above query on plcs1, peft2, peft3 and prsc1k

  The script on lcs2ksc0 runs the above query on plcs2k, peft2, peft3 and prsc1k

 

If the script running on either LCS server detects an error or a different result for the above query, an error message is sent to Tivoli and RSC Support is paged.

RSC Support is the first point of contact when this script sends an error to Tivoli. RSC Support may then contact the DBA to help with troubleshooting the issue or in checking whether or not there is a problem with replication.

Maintenance Windows

The standard maintenance window for the LCS is 9am to 12pm on Tuesday mornings on those Tuesdays where there is an approved window. Typically about one Tuesday per month is an approved window. Approved change windows for the APC are available from UNIX Operations or Phillip Sherrell.

It is possible for changes to be carried out on other days of the week other than Tuesdays. The CCF process is used to approve such changes. The change window of 9am-12am is persistent regardless of the day of the week. All changes on LCS must be confined to 9am-12pm and no changes outside this 3 hour window are ever permitted because Draft Capture Files and other reporting functions are running outside this window.

Times of the day that are absolutely critical for LCS are:

  Midnight to 6am daily. This is when Core Reports RETIRED is run. Daily Partition Maintenance jobs also run at this time, as do stats gathering jobs.

  5:30pm-4:30am daily. This is when many Draft Capture files are running, although Draft Capture files run at other times of day as well.

  12:15pm daily. This is when the first daily run of prod_fix RETIRED takes place. prod_fix runs at 2 hours intervals from 12:15 to midnight daily.

  The first day of the month. Monthly Billing Reports run on this day.

  The 4th day of the month. The GST reports RETIRED run on this day.

  The first 10 days of the month. The Usage Reports are running during this time.

Procedures for Taking a Planned Database Outage

Shutting Down

If an LCS production database needs to be taken down for maintenance, here is what needs to be done before you can shutdown the database.

  Raise a GSM ticket.

  Ensure the maintenance is being carried out during an approved maintenance window.

  Ensure no RMAN backup is currently running or scheduled to run during the outage.

  The LCS database to taken down must not be configured as master. An LCS failover is needed if the database requiring work is currently the master. After the failover, the master will be a different box and this one will then be available to be taken down for maintenance. The OCC can tell you which LCS server is master at any one time.

  mc_txnblob_read needs to be stopped by UNIX Operations. As well as stopping the actual process, UNIX Operations need to stop the Tivoli monitor for mc_txnblob_read as well, otherwise Tivoli will restart mc_txnblob_read as soon as it is shut down. See the mc_txnblob_read section for details.

  Stop the Streams propagation processes on each RSC and MIGS database that propagate to this LCS database. See RSC and LCS Replication shutdown and startup procedures.

  Stop the DCF Scheduler and Aristion Extract or else have UNIX Operations or UTS do this:

sesudo lcs_app

cd /apps_01/lcrpts/script

./dcfctl stop

./AristionCtl stop dest=file

./AristionCtl stop dest=table

  Stop KitMon:

sesudo ats_db

cd /apps_01/ats_db/rtm/shell

./rtmctl stop

  Inform MIGS Fraud Monitoring group (Paul Edwards and Michael Leung) as connectivity from the EMS application will be interrupted.

  Have UTS carry out their procedures to stop LCS and ATS applications. For the ATS application they may need to kill any lingering processes that are running as the ats_app user.

  If any applications are still running such as Usage Reports, GST Reports, RETIRED Virgin Blue extract RETIRED or anything else then you can terminate these by doing a SHUTDOWN IMMEDIATE of the database but these applications will need to be restarted after the database comes back up.

  You can then take down the database as per DBA procedures. These include stopping the OEM agent and any other DBA infrastructure that needs to be taken down including the database. I make no attempt to document this here please consult the applicable DA standard documentation.

Starting Up

After the database maintenance and once the database and standard database infrastructure have been started here are specific steps needing to be done in the LCS environment:

  Start any stopped Streams propagation processes on each RSC and MIGS databases. See RSC Replication shutdown and startup procedures.

  Have UNIX Operations start mc_txnblob_read and its Tivoli monitor. Have them make sure both the process and the Tivoli monitor are running.

  Do not rerun any failed RMAN backup and dont release held backup jobs. Just let the backup job run as normal when it is next scheduled.

  Inform MIGS Fraud Monitoring group (Paul Edwards and Michael Leung) as they may need to bounce the EMS application.

  Start KitMon:

sesudo ats_db

cd /apps_01/ats_db/rtm/shell

./rtmctl start

  Start the DCF Scheduler and Aristion Extract or else have UNIX Operations or UTS do this:

sesudo lcs_app

cd /apps_01/lcrpts/script

./dcfctl start

./AristionCtl start dest=file

./AristionCtl start dest=table

  Have UTS carry out their procedures to start LCS and ATS applications.

  Have RSC Support start any reports or processes that were terminated when shutting down.

  If the Usage Reports were running at the time of the shutdown then rerun them:

sesudo ats_db

nohup /apps_01/ats_db/bin/runUsageReport.sh -s plcs1 >> /apps_data_01/ats_db/log/runUsageReport.log 2>&1 &

LCS Release Implementation

No more LCS releases are scheduled. Retirement of the LCS is imminent!

There are typically two major releases of the LCS every year and these releases are part of the RSC release cycle. These releases are given the name yy.n to denote the release, for example:

08.2 2nd release of 2008

09.1 1st release of 2009

09.2 2nd release of 2009

etc

The yy.1 release is normally implemented in April-May of the calendar year.

The yy.2 release is normally implemented in September-October of the calendar year.

The releases are driven and coordinated by RSC Support.

The APC DBA normally incorporates major database changes such as version upgrades, patches, major audit initiatives, major tuning and database reorganisation initiatives etc into one these releases. These releases may also include schema changes for the application as required by the LCS applications.

Data Elements and Table Columns

Much of the data in the RSC is ISO standard financial data. Each such data element is given an ISO standard name which is recognised industry wide. Examples of such names are:

DE002 is the card number

DE032 is the acquiring institution id

DE039 is the response code

DE042 is the merchant identifier

etc

I have a spreadsheet which shows the mapping of data element ids to table columns in the RSC, LCS and MIGS, where known. You can access this spreadsheet here.

Matching MIGS Transactions in the LCS

MIGS is a front end to the RSC so all transactions in MIGS go through to the RSC and therefore subsequently to LCS. The only MIGS transactions which dont appear in the RSC are test transactions. It is possible to match MIGS transactions with RSC/LCS transactions using certain criteria. I have documented how to identify a MIGS transaction in the LCS in a separate document which you can access here.

The Future of the LCS

The RSC will be retired by the end of 2011. Once the RSC retires, the LCS will no longer have data feeding it, so it will wither away and die. Essential reporting functions that are currently being done by LCS will need to be done by other systems. I have written document called The Future of the APC, which details current thinking of how LCS functions will be performed in the future, once the RSC is retired.
